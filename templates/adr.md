# Architecture Decision Record — Template

One file per consequential decision, saved at `docs/08-architecture/adr/ADR-XXX.md` in the target project (see `rules/document-locations.md`). Created whenever a playbook (typically 08-Architecture, but any phase may produce one for a decision with a high cost of reversal — see `rules/confirmation-protocol.md`'s always-strict categories) confirms a decision worth recording for future reference. Follows the heading + metadata-line convention in `rules/document-format.md`, same as every other document.

## Structure (Casual)

```markdown
# ADR-001 — <short, specific title — e.g. "Use PostgreSQL as the primary datastore">
*Status: Accepted · Date: 2026-07-13 · Traces to: REQ-012*

## Context
<What situation/problem/requirement prompted this decision? Reference the
REQ-XXX or discussion that led here — do not restate the whole requirement,
just enough context for the decision below to make sense on its own.>

## Decision
<The decision, stated as a single clear sentence. Not a discussion — the
conclusion.>

## Alternatives considered
<Every real alternative that was on the table, one bullet each, with the
one-line reason it was rejected. If there was truly only one option, say so
explicitly — do not fabricate alternatives to fill this section.>

## Consequences
<What this decision makes easier, what it makes harder, and what it
forecloses. Both positive and negative — a decision with no stated
downside was not actually analyzed.>
```

`Status` is `Proposed` / `Accepted` / `Superseded` / `Deprecated`. `Traces to` holds the `REQ-XXX` (non-functional requirements this decision addresses) and/or `ARCH-XXX` (the component/decision this ADR documents), per `rules/traceability-rules.md`. When `Status: Superseded`, add a `Superseded by` key naming the new `ADR-XXX` — the superseded record is never deleted, only marked, per `rules/id-conventions.md`.

## Fully Dressed additions

```markdown
# ADR-001 — <title>
*Status: Accepted · Date: 2026-07-13 · Traces to: REQ-012*

## Decision drivers
<The specific forces that shaped this decision — not the full Context,
just the factors that actually pushed the choice one way: a hard
constraint, a team's existing expertise, a cost ceiling, a compliance
requirement. What a reader needs to judge whether the decision still
holds if one of these drivers changes.>

## Context
...

## Decision
...

## Alternatives considered
...

## Consequences
...

## Related decisions
<Other ADR-XXX this one builds on, conflicts with, or is superseded
alongside — "(none)" if genuinely standalone.>
```

## Notes for whoever fills this in

- The `Context` and `Decision` sections must reflect what the user actually confirmed (`rules/confirmation-protocol.md`) — never a rationale the AI constructed after the fact to justify a choice the user didn't explicitly make.
- If a later Change Request (`rules/change-management.md`) changes this decision, do not edit this file's `Decision` in place — create a new ADR, mark this one `Status: Superseded`, and set `Superseded by`.
