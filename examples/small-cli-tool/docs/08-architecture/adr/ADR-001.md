## Context
REQ-002 requires converting files up to 100MB while staying under 512MB of memory. The tool also has no need for a server component — it's a single-developer, single-invocation utility (see Vision, "constraints": no runtime to install beyond what's already there).

## Decision
Build a single-process CLI binary that streams input records through the Conversion Engine (ARCH-001) rather than loading the whole file into memory, and that has no client-server architecture at all.

## Alternatives considered
- **Load whole file into memory, simpler implementation**: rejected — a 100MB CSV parsed into JS objects can easily exceed 512MB, especially with nested-object flattening.
- **A small local HTTP server the CLI talks to**: rejected — adds complexity and a runtime dependency (a listening port) for no benefit; there's no multi-client or remote-access need.

## Consequences
Streaming parsing is more complex to implement than "read the whole file, transform, write" — but it's the only approach that reliably satisfies REQ-002 for large files. Testing (phase 12) needs a large-file test case to actually verify the memory bound, not just correctness on small inputs.
