## Context
The team is small (a handful of engineers) and the Vision sets a one-quarter MVP deadline. The product will eventually need to scale across many tenants, but doesn't yet have the team size or operational maturity microservices assume.

## Decision
Build a modular monolith: one deployable Application Server (ARCH-001), internally organized into modules by domain (projects, tasks, users) with clear boundaries, so a future split into services remains possible without a full rewrite.

## Alternatives considered
- **Microservices from day one**: rejected — the operational overhead (service discovery, distributed tracing, multiple deploy pipelines) isn't justified by the current team size or the MVP deadline, and would likely slow the first release rather than speed later ones.
- **Unstructured monolith (no internal module boundaries)**: rejected — would make a later split much harder and encourages the kind of tangled cross-domain logic that's expensive to unwind.

## Consequences
Faster initial development and a single deploy pipeline, at the cost of needing deliberate internal discipline to keep module boundaries clean — this is a process cost, not a technical one, and needs to be enforced in code review, not just documented here.
