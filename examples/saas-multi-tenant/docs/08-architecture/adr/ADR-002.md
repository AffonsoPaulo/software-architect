# ADR-002 — Tenant isolation enforced by middleware-injected scoping, not per-tenant databases
*Status: Accepted · Date: 2026-05-12 · Traces to: REQ-003 · Author: Priya*

## Decision drivers
- REQ-003's zero-tolerance requirement: no cross-tenant access under any circumstance.
- Operational cost at the target scale (10 customer organizations in year one) — must not require per-tenant operational overhead that doesn't pay for itself yet.
- Defense in depth: a single enforcement layer means a single point of failure for the product's most consequential requirement.

## Context
REQ-003 requires that no tenant can ever access another tenant's data, including via crafted requests. Two common approaches exist: a separate database (or schema) per tenant, or a shared database with mandatory tenant scoping enforced at the application/query layer.

## Decision
Use a shared database with tenant scoping enforced by the Tenant Context Middleware (ARCH-002) on every query, backed by PostgreSQL row-level security policies as a second, independent enforcement layer — so a bug in application-level scoping doesn't silently defeat isolation.

## Alternatives considered
- **Database-per-tenant**: rejected for the MVP — operationally heavier (migrations must run per tenant, connection management scales with tenant count) for a benefit (isolation) that shared-DB-plus-RLS already provides at this scale. Revisit if a specific enterprise customer requires physical data separation as a contractual term.
- **Application-layer scoping only, no RLS**: rejected — a single missed `WHERE tenant_id = ?` in a new query would be a full tenant-isolation breach with no second layer to catch it. RLS makes that class of bug fail closed instead of silently leaking data.

## Consequences
Every new table needs an RLS policy defined alongside its schema — this is now a required step in Database Design (phase 07) for any future table, not optional. Testing (phase 12) needs an automated cross-tenant-access test that exercises this specifically (TEST-003), not just unit tests of the middleware in isolation.

## Related decisions
Builds on ADR-001 — both enforcement layers live inside the single modular monolith rather than a separate service.
