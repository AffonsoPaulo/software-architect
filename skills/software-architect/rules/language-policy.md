# Language Policy

Two independent decisions ‚Äî never conflate them.

## 1. The Skill's own content: always English

Every file inside `software-architect/` ‚Äî `SKILL.md`, every `playbooks/`, `rules/`, `templates/`, `checklists/`, `quality-gates/`, `scripts/`, and `docs/` file ‚Äî is written in English, fixed, regardless of who uses the Skill or what language they converse in. This keeps the Skill compatible with the broader `npx skills` ecosystem and usable/contributable-to by anyone, not only the original author.

## 2. Documents the Skill generates: the target project's language

Documents the Skill writes into the *user's* project (Vision, Requirements, Architecture, and so on ‚Äî see `document-locations.md`) follow whatever language the user is conversing in during the interview. This is:

- **Detected**, not assumed, during `playbooks/00-project-calibration.md` ‚Äî the AI notices the language the user is writing in and proposes it, but confirms explicitly before locking it in (`[confirmation individual]`). These are two different things with two different timings: the AI's own conversational responses switch to match the language the user just used immediately, in that same turn ‚Äî noticing the switch without actually speaking it back is not sufficient, and asking the confirmation question itself should already happen in the user's language. Only the `language` field that governs generated documents waits for the explicit `[confirmation individual]` answer before being locked in.
- **Stored once**, in `project-state.md`'s `language` field, and reused for every subsequent phase ‚Äî never re-asked per phase, never silently re-inferred.
- **Project-wide**, not per-cycle ‚Äî a language set in cycle 1 applies to cycle 2 unless explicitly changed (see below).

**"The document follows the confirmed language" means the whole document ‚Äî its structural headings and section labels included, not only the user's own free-text answers.** A template's `# Vision`, `## Business processes`, `## Actors`, and so on are the Skill's own English reference structure (rule 1 above) ‚Äî the AI writes the *actual* document with those same sections, in the same order, but phrased in the project's confirmed language (`# Vis√£o`, `## Processos de Neg√≥cio`, `## Atores`, for a `language: "pt"` project). A document that's entirely in Portuguese except for an English `# Vision` at the top fails `rules/document-format.md`'s "reads like a native document, pasteable into Word" bar just as much as a document with untranslated prose would. `scripts/validate-heading-language.mjs` checks this mechanically ‚Äî every phase's fixed section headings, checked against the exact English text each template shows, whenever the project's `language` isn't `"en"` ‚Äî because this specific instruction has been missed in practice even when stated exactly this explicitly.

**A table's own column headers are structural labels too, exactly like a heading** ‚Äî both the named-column tables (`templates/use-cases.md`'s `| Step | Condition | Result |`, `templates/vision.md`'s `| Stakeholder group | Cares about | Success looks like |`) and the `| Field | Value |` shell tables (`rules/document-format.md`'s "Two shapes of table"). A project confirmed as `language: "pt"` with a translated `## Perfis de stakeholders` heading sitting directly above an untranslated `| Stakeholder group | Cares about | Success looks like |` row has the exact same problem as an untranslated heading ‚Äî a template showing that row as ready-to-copy example markdown is not license to leave it in English. This is distinct from a table's *data*: a fixed enum value inside a cell (a CR's `Status` column reading `Pending reapproval`, `calibration.md`'s `included`/`skipped`) stays untranslated the same way a metadata-line key does, since it's schema, not prose ‚Äî only the column header itself is what a stakeholder actually reads. `scripts/validate-heading-language.mjs` checks known table headers the same mechanical way it checks section headings.

This applies only to prose a stakeholder actually reads. It never extends to:
- **Folder and file paths** (`docs/00-calibration/`, `calibration.md`, `req-001.md`) ‚Äî these are addressing, not content, and stay exactly as `rules/document-locations.md` defines them regardless of project language. Scripts (`scripts/build-doc-site.mjs`'s phase table, every `validate-*.mjs`) hardcode these paths; translating them would require the whole toolchain to special-case every language a project might use, for a part of the file no stakeholder ever reads as prose.
- **`project-state.md` itself** ‚Äî it stays English, full stop, per its own exemption in `rules/document-format.md`: it's the Skill's operational state, never stakeholder-facing, so field names *and* values (`name: "project-calibration"`, `status: "completed"`) are fixed regardless of the project's `language`.
- **Metadata-line keys** (`Traces to`, `Delivers`, `Depends on`, and any other key `scripts/lib/docs.mjs` reads ‚Äî `rules/document-format.md`'s "Standard keys" table) ‚Äî the key itself stays in English exactly as that table spells it (`*Traces to: BR-002*`, never `*Rastreia at√©: BR-002*`); only the *value* after the colon, and any other key not in that table, follows the project's language. This is easy to miss, since the metadata line otherwise reads like ordinary translated prose (a "subtitle or caption," per `rules/document-format.md`) sitting right under a translated heading ‚Äî but the key is what `scripts/lib/docs.mjs` matches literally (`LIST_KEYS`) to build the whole traceability graph. A translated key isn't a smaller-scope error the same way a mistranslated heading is: it makes that artifact's entire reference invisible to every validation script silently ‚Äî no orphan reported, no broken-link warning, nothing ‚Äî since the graph never saw the edge to begin with.
- **A metadata-line value drawn from a template's own fixed, finite vocabulary** ‚Äî `Type: Functional`/`Non-functional` (`templates/requirements.md`), `Kind: Automated`/`Manual` (`templates/testing.md`) ‚Äî stays in English too, the same as the key it follows, not because the *value* position is special but because these two specific values are schema a script matches literally (`scripts/validate-traceability.mjs`'s functional/non-functional coverage split, `scripts/validate-gate.mjs`'s Testing-kind and Requirements-type checks), not prose a stakeholder reads for its wording. A genuinely free-text value in the same position ‚Äî `Source`, `Owner`, a `Target date` ‚Äî follows the project's language normally; only these specific closed-vocabulary values don't. Same silent-failure shape as a translated key: a translated `Type` value doesn't error, it just makes that requirement invisible to the functional/non-functional coverage split ‚Äî both scripts now flag an unrecognized `Type` explicitly instead of silently skipping it, but the value itself should never reach that state to begin with.

## Changing language mid-project

If the user switches language mid-project, this is treated as a normal correction via `confirmation-protocol.md` ‚Äî not a formal Change Request (`change-management.md`). It does not retroactively translate or affect the content of already-approved documents; it only applies to documents written from that point forward.

## Why these two decisions are independent

A team could run the Skill entirely in Portuguese for their project's documents while the Skill's own instructional files stay in English ‚Äî the two never need to match, and nothing in the Skill's logic assumes they do.
