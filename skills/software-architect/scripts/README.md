# scripts/

Plain Node.js (built-in modules only, no npm dependencies — `npx skills` already requires Node, so this avoids a second runtime dependency). Validation scripts, one presentation tool, and one file-mutating tool (`new-cycle.mjs` — every other script here is read-only).

- `validate-ids.mjs` — checks artifact ID format and uniqueness against `rules/id-conventions.md`
- `validate-traceability.mjs` — builds the traceability graph from `rules/traceability-rules.md` and reports orphans/broken references, respecting phases marked as skipped in the target project's `project-state.md`
- `validate-versioning.mjs` — checks `rules/versioning.md`'s mechanism: every cycle and every artifact has a non-blank `Author`; `docs_version` is well-formed and matches `docs/CHANGELOG.md`'s newest entry; `project-state.md`'s `changelog[]` and `CHANGELOG.md` never drift apart; every changelog entry references a real cycle or Change Request
- `validate-gate.mjs` — generic runner that loads a phase's `quality-gates/` + `checklists/` pair and reports which scriptable criteria pass
- `audit-compatibility.mjs` — implements `rules/skill-drift.md`'s compatibility audit: reruns `validate-gate.mjs` for every phase whose current status is `completed` (across all cycles, so a phase reopened by a later incremental cycle is picked up too) against the *currently installed* Skill's rules, and reports what's now missing — read-only, never modifies anything, never decides a gate passed/failed on its own (judgment criteria always need a human, same posture as `validate-gate.mjs`). Run it yourself with `node scripts/audit-compatibility.mjs [project-root]` whenever the Skill's version has advanced since a project (or an earlier phase in it) was last touched.
- `new-cycle.mjs` — used by `playbooks/00-project-calibration.md`'s incremental mode: appends a new cycle to `project-state.md`'s `cycles[]` and repoints `active_cycle_id`, computing the next id and initializing every field consistently, via a targeted text insertion rather than a full parse-rewrite (so the file's inline comments survive). Only runs once scope and author are already confirmed conversationally — it doesn't ask or decide anything, and refuses to run if the current active cycle hasn't reached `ready_for_implementation: true`. `node scripts/new-cycle.mjs <project-root> --scope "<scope>" --author "<author>"`
- `self-test.mjs` — regression test: runs the validators above against both projects in `examples/` and fails if either isn't clean. `examples/` only exists on the `with-examples` branch (see root `README.md`) — on `main` this script reports both as intentionally skipped.
- `build-doc-site.mjs` — not a validator: builds a single, self-contained HTML page from a target project's `docs/` — a clickable table of contents plus every document in reading order, offline (Mermaid is vendored, not loaded from a CDN). Read-only, never runs automatically; run it yourself with `node scripts/build-doc-site.mjs [project-root] [output-path]` whenever you want a browsable view instead of reading file by file.
- `lib/` — shared parsers (markdown heading+metadata-line extraction, YAML for `project-state.md`, document indexing) and `lib/vendor/` (the vendored Mermaid.js build)
