# scripts/

Plain Node.js (built-in modules only, no npm dependencies — `npx skills` already requires Node, so this avoids a second runtime dependency). Validation scripts, one presentation tool, and one file-mutating tool (`new-cycle.mjs` — every other script here is read-only).

- `validate-ids.mjs` — checks artifact ID format and uniqueness against `rules/id-conventions.md`
- `validate-traceability.mjs` — builds the traceability graph from `rules/traceability-rules.md` and reports orphans/broken references, respecting phases marked as skipped in the target project's `project-state.md`
- `validate-versioning.mjs` — checks `rules/versioning.md`'s mechanism: every cycle and every artifact has a non-blank `Author`; `docs_version` is well-formed and matches `docs/CHANGELOG.md`'s newest entry; `project-state.md`'s `changelog[]` and `CHANGELOG.md` never drift apart; every changelog entry references a real cycle or Change Request
- `validate-tone.mjs` — checks `rules/document-format.md`'s "Never let the Skill's own process show through": no document (other than `project-state.md`, which is exempt) references this Skill's internal phase numbers ("Phase 08"), leaves a `[confirmation individual]`-shaped bracket marker in written text, cites a `rules/`/`playbooks/`/`templates/`/`scripts/` path, or echoes the Skill's own policy language ("always mandatory") as if it were the project's own reasoning
- `validate-heading-language.mjs` — checks `rules/language-policy.md`'s requirement that structural headings and table column headers follow the project's confirmed language, not just prose: for any project where `language` isn't `"en"`, no phase document's fixed section headings (a template's own `## Structure` example, including phase 17's Review Report and every `docs/change-requests/*.md`) and no known table header row (`Field`/`Value`, `Step`/`Condition`/`Result`, and so on) are still the template's literal English text. A no-op when the project's language is English.
- `validate-gate.mjs` — generic runner that loads a phase's `quality-gates/` + `checklists/` pair and reports which scriptable criteria pass, plus a per-phase mechanical check for the criteria that don't fit the generic ID/traceability model: Calibration's project-state.md/calibration.md structural completeness (00), Vision's expected sections and Fully Dressed tables (01), Requirements' acceptance criteria/edge cases (03), Use Cases' postconditions (05), Domain Model's invariants/aggregate membership (06), Testing's explicit Automated/Manual kind (12), Roadmap's milestone dependency cycles (14, alongside its existing coverage check), and Implementation Plan's Definition of Done (16, alongside its existing task-dependency cycle check)
- `audit-compatibility.mjs` — implements `rules/skill-drift.md`'s compatibility audit: reruns `validate-gate.mjs` for every phase whose current status is `completed` (across all cycles, so a phase reopened by a later incremental cycle is picked up too) against the *currently installed* Skill's rules, and reports what's now missing — read-only, never modifies anything, never decides a gate passed/failed on its own (judgment criteria always need a human, same posture as `validate-gate.mjs`). Run it yourself with `node scripts/audit-compatibility.mjs [project-root]` whenever the Skill's version has advanced since a project (or an earlier phase in it) was last touched.
- `new-cycle.mjs` — used by `playbooks/00-project-calibration.md`'s incremental mode: appends a new cycle to `project-state.md`'s `cycles[]` and repoints `active_cycle_id`, computing the next id and initializing every field consistently, via a targeted text insertion rather than a full parse-rewrite (so the file's inline comments survive). Only runs once scope and author are already confirmed conversationally — it doesn't ask or decide anything, and refuses to run if the current active cycle hasn't reached `ready_for_implementation: true`. `node scripts/new-cycle.mjs <project-root> --scope "<scope>" --author "<author>"`
- `self-test.mjs` — regression test: runs the validators above against both projects in `examples/` and fails if either isn't clean
- `build-doc-site.mjs` — not a validator: builds a single, self-contained HTML page from a target project's `docs/` — a clickable table of contents plus every document in reading order, offline (Mermaid is vendored, not loaded from a CDN). Read-only, never runs automatically; run it yourself with `node scripts/build-doc-site.mjs [project-root] [output-path]` whenever you want a browsable view instead of reading file by file.
- `build-doc-word.mjs` — the Word counterpart: builds a single `.rtf` from the same `docs/` tree, opening directly in Word, LibreOffice, Pages, or Google Docs' importer — no npm dependency, since RTF is plain text (no ZIP/OOXML container to hand-build the way a real `.docx` would need). Headings carry outline levels and a real Word heading style, so the Navigation Pane and Insert > Table of Contents both work on the file directly — no hand-rolled sidebar needed. Artifact cross-references (`[REQ-001](req-001.md)`, the shape index tables actually use) become real, clickable hyperlinks to a bookmark on that artifact's own heading; a link this script can't resolve to a known artifact ID renders as plain text rather than a broken jump. Mermaid code fences render as a labeled, monospaced source block with a note pointing at mermaid.live (`project-state.md`'s own translated `export_labels.mermaid_fallback`, same as the Table of Contents/Change Requests chrome) — RTF has no script execution to render a diagram the way the HTML build does, the same documented fallback `rules/document-format.md` already accepts for tools that can't render Mermaid natively. Read-only, never runs automatically; run it yourself with `node scripts/build-doc-word.mjs [project-root] [output-path]`.
- `lib/` — shared parsers (markdown heading+metadata-line extraction, YAML for `project-state.md`, document indexing, RTF text/field escaping, the Markdown-to-RTF renderer) and `lib/vendor/` (the vendored Mermaid.js build)
